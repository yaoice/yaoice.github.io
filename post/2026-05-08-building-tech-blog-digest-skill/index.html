从一次性脚本到可评测 Skill：tech-blog-digest 打造全过程

把每周一次的开发者博客整理工作，做成可复用的 Skill

Posted by 爱折腾的工程师 on Friday, May 8, 2026

引言

每周维护博客的固定动作：抓 Google Developers Blog、OpenAI Developers Blog、Claude Blog 三家最新文章 → 选 1~3 篇 → 整理成中文图文落到 Hugo 仓库（content/post/<slug>.md + static/img/<slug>/*.svg）→ 跑 hugo --gc --minify 验证。

最初是一次性脚本：curl + BeautifulSoup + 手写 Python 拼一篇，每次大约 7 分钟。重复了 6 次后这条流水线被沉淀成一个 Skill：tech-blog-digest。本文记录的是它的工程化过程——从设计、实现，到对它做 A/B 评测，再到根据评测结论反过来修 Skill 自身。

涉及的目录约定（CodeBuddy IDE 的 Skill 规范）：

.codebuddy/skills/tech-blog-digest/
├── SKILL.md                          # 触发条件 + 工作流文档
├── scripts/
│   ├── _common.py                    # locate_repo_root() 等共用工具
│   ├── fetch_listing.py              # 抓三家列表页 → JSON
│   ├── fetch_article.py              # 抓单篇正文 + 下图 → JSON
│   ├── render_post.py                # body.md + meta.json → Hugo 博文
│   └── requirements.txt              # requests + beautifulsoup4
├── references/
│   ├── post_template.md              # 中文复述写作模板
│   └── sites.md                      # 三家站点选择器排错手册
└── state/
    └── seen.json                     # 已抓 URL 增量去重

一、问题域：6 条隐性领域知识

把"整理一篇外文博客到中文 Hugo 仓库"拆开，至少 6 条领域知识藏在动作下面，每一条都是 baseline agent 容易踩坑的点：

① 各站点正文容器选择器不一致

# 分别试过多个候选才稳定
SOURCE_CONTAINERS = {
    "google":  ["article", "main article", "div.post-content"],
    "openai":  ["article", "main article", "main"],
    "claude":  ["div.blog-post_rich-text", "article", "div[class*='rich-text']"],
}

Claude 是 Webflow 站，正文藏在 div.blog-post_rich-text 里——这个 class 名不读 HTML 看不出来。

② 模板垃圾要清洗

抓到的正文末尾普遍带这种东西：

...
No items found.
Prev   0 / 5   Next
eBook   FAQ
Related posts
Subscribe   Contact sales

fetch_article.py 用一个 clean_main() 把它们干掉：

REMOVE_TAGS = {"script","style","noscript","iframe","form",
               "nav","footer","aside","header","button","svg"}

def clean_main(main):
    for tag in REMOVE_TAGS:
        for el in main.find_all(tag):
            el.decompose()
    for el in main.find_all(attrs={"class": re.compile(
        r"(share|subscribe|related|newsletter|comment)", re.I)}):
        el.decompose()
    return main

baseline agent 没做这步，残留的"Subscribe / Contact sales"会直接进入中文博文。

③ 站点品牌后缀

og:title 长这样：

Deploying Claude across financial services | Claude

要 strip 尾巴：

meta["title"] = re.sub(
    r"\s*[|\-\\—–]\s*(OpenAI Developers|Google Developers Blog|Anthropic|Claude)\s*$",
    "", meta["title"]).strip()

④ 图片下载 + 改写 src 要可重跑

每张图按 {idx:02d}-{hash前缀}.{ext} 命名，避免同样输入不同跑产生不同文件名：

h = hashlib.sha1(r.content).hexdigest()[:8]
fname = f"{idx:02d}-{h}{ext}"          # 例: 01-f6bbebe8.svg
(img_dir / fname).write_bytes(r.content)
el["src"] = f"/img/{slug}/{fname}"
if el.has_attr("srcset"):
    del el["srcset"]                   # 否则 Hugo 渲染会走外链

⑤ Hugo front matter 字段对齐仓库历史风格

仓库 content/post/*.md 的字段是 title/date/draft/tags/categories/description/image/source/origin_url——baseline agent 会照新的 .md 一份份去看才能模仿对，render_post.py 直接把它写死。

⑥ 增量去重

state/seen.json 是个 URL 集合：

$ python3 fetch_listing.py --sources claude --limit 1 --mark-seen
$ cat state/seen.json
[
  "https://claude.com/blog/deploying-claude-across-financial-services"
]

下次再跑 fetch_listing.py，集合内的 URL 自动从结果中剔除。baseline 没这层状态，每次重跑都得人工剔重。

二、7 步骨架：脚本动作 vs agent 决断

7 步分两类：5 个脚本动作（蓝框）零判断空间，2 个agent 决断（橙框）模型必须参与。

数据流

整条流水线的中间数据全是 JSON / Markdown，靠文件系统串起来：

fetch_listing.py
   └── stdout JSON  →  /tmp/listing.json
                       │
                       ▼
                 [agent 列摘要表，等用户回复]
                       │
                       ▼
fetch_article.py URL --slug claude-xxx
   └── stdout JSON  →  /tmp/<slug>.json
                       │
                       ▼
                 [agent 写中文复述]  →  /tmp/<slug>.md
                       │
                       ▼
render_post.py --meta /tmp/<slug>.json --body /tmp/<slug>.md
   └── 落盘  →  content/post/<slug>.md
                       │
                       ▼
hugo --gc --minify
                       │
                       ▼
fetch_listing.py --sources <选中> --mark-seen

Step 1：拉列表（脚本）

fetch_listing.py 抓三家列表页，stdout 输出 JSON：

{
  "fetched_at": "2026-05-06T15:46:51",
  "limit_per_source": 1,
  "result": {
    "claude": [
      {
        "source": "claude",
        "title": "Deploying Claude across financial services",
        "url": "https://claude.com/blog/deploying-claude-across-financial-services",
        "date": "May 5, 2026",
        "summary": ""
      }
    ]
  }
}

设计上单源失败不致命：抛异常被捕获后写入 errors[source]，其它源继续。--sources google,openai,claude 子集 + --no-dedup + --mark-seen 三个开关覆盖了"抓全/抓单/重新抓/标已读"四种场景。

Step 2：摘要确认（agent 决断）

脚本不能替用户做选择题。agent 把 fetch_listing.py 的 JSON 渲染成 Markdown 表格给用户看：

| # | 来源 | 日期 | 标题 | 一句话摘要 | 备选 slug |

然后等用户回复编号（如 1,3,5）。SKILL.md 用"禁止跳过“和"未得到回复前不要进入 Step 3"两条硬约束限定 agent 行为。这也是 iteration-2 要修的冲突点——后面第四节会讲。

Step 3：抓正文 + 下图（脚本）

fetch_article.py <url> --slug <slug> 抓单篇详情页，输出 JSON 含 content_html / content_text / images[] / cover_image / tags / published。关键动作是 download_images 把每个 <img> 的 src 改写为本地 /img/<slug>/...——这一步是图片本地化的关键，agent 后续写 markdown 时直接引用本地路径就行，不用关心原 URL。

Step 4：写中文复述（agent 决断）

模型最擅长、脚本最做不了的部分。references/post_template.md 给写作结构（导读 + 背景 + 核心做法 + 我的理解 + 参考链接），不限制具体措辞。agent 基于 content_text 写完后保存到 /tmp/<slug>.md。

Step 5：render_post.py（脚本）

把 meta.json + body.md 拼成完整 Hugo 博文落盘到 content/post/<slug>.md。重点在 to_hugo_date：

def to_hugo_date(s: str) -> str:
    dt = None
    if s:
        for fmt in ("%Y-%m-%dT%H:%M:%S%z", "%Y-%m-%dT%H:%M:%SZ",
                    "%Y-%m-%dT%H:%M:%S.%fZ", "%Y-%m-%d",
                    "%B %d, %Y", "%b %d, %Y"):
            try:
                dt = datetime.strptime(s, fmt); break
            except ValueError:
                continue
    if dt is None:
        dt = datetime.now(timezone.utc)        # ← 关键：空串会 fallback
    ...

这个 fallback 是后面 iteration-2 要修的隐性 bug——参数 s 经常是空串（OG meta 不全），fallback 后 date 就被错记成"运行那天”。

Step 6：hugo build 验证（脚本）

hugo --gc --minify

退出码 0 才进入 Step 7。最常见失败原因是 front matter 字段里的双引号未转义，或 cover_image 路径不存在（图片下载全失败时会留空）。

Step 7：mark-seen（脚本）

python3 fetch_listing.py --sources <选中> --limit <渲染数> --mark-seen

把成功入仓的 URL 写入 state/seen.json，下次 fetch_listing.py 自动剔除。

注意这里不是简单重跑 Step 1 的命令——如果用户只挑了部分文章，窄化 mark-seen 的范围非常关键，不然会悄悄丢失未圈选的文章。这是后面 iteration-2 要修的第三个隐性约束。

三、A/B 评测：8 条机检断言 + transcript 对照

写完 Skill 不算完——要回答"它是否真的有价值"。

评测设计

  • 同一个 prompt：“抓 Claude Blog 最新 1 篇，整理成中文图文落到 Hugo 仓库（不用问我）。”
  • 两条管道并行：with-skill 调用 SKILL.md 的 7 步流程；without-skill 完全不接触 .codebuddy/skills/，从零裸做。
  • 隔离副本：rsync 一份仓库到 /tmp/tbd-eval/，避免污染主仓库。
  • 8 条机检断言：
ASSERTIONS = [
    "post-md-exists",       # content/post/ 下产出至少 1 篇 .md
    "front-matter-valid",   # 含 title 和 date
    "no-mermaid-block",     # 不含 ```mermaid 代码块
    "no-remote-img",        # 图片均为本地 /img/ 路径
    "img-files-exist",      # 引用图片在 static/ 真实存在
    "source-is-claude",     # 来源域名出现在博文中
    "hugo-build-ok",        # hugo --gc --minify 退出码 0
    "exactly-one-post",     # 仅 1 篇
]

实际耗时 ~3 min ~7 min

客观指标的天花板

8/8 vs 8/8 之所以打平，原因有两个层面：

  1. 断言抓硬底线：“产物形式合规"是个低门槛——baseline 在熟练 agent 手里勉强能撞过去。
  2. 熟练 agent 知道这个仓库：我作为主 agent 提前知道 Webflow 用 div.blog-post_rich-text、知道仓库 front matter 字段、知道 CODEBUDDY.md 的 mermaid 禁令。换个不熟悉这个仓库的 agent 来跑 baseline，第②~⑤条断言一半会崩。

结论：Skill 的真实价值是执行下限——把"做对这件事所需的全部领域知识"封装下来，让任何 agent 照做都能跑出同等质量的产物。这个价值客观断言抓不到，只能靠 transcript 对照看。

四、iteration-2：从"agent 必须聪明"到"文档已写明”

iteration-1 双跑虽然机检打平，但跑过程中我作为主 agent 做了 3 个SKILL.md 没明文要求的关键动作。换个不熟悉这个 Skill 的 agent 来，每一处都可能崩。

三个隐性约束

① Step 2 与"用户显式豁免"的冲突

SKILL.md（修订前）：

### Step 2：摘要确认（**禁止跳过**）
...
未得到用户回复前，**不要**进入 Step 3。

但 prompt 里有"不用问我，直接整理"。这是个硬约束 vs 用户显式意图的冲突，agent 没标准答案。我自己做的处理是"跳过反向提问 + 在最终汇报里列选中清单"——但 SKILL.md 没出口条款。

② Step 4 published 字段丢失

复现：

>>> art = json.load(open('/tmp/article.json'))
>>> art['published']
''                                 # ← 空串
>>> # 直接喂给 render_post.py
>>> render_post.to_hugo_date('')
'2026-05-06T07:46:00Z'             # ← 当前 UTC 时间！

问题在 fetch_article.py 解析 og meta：

pub = soup.find("meta", property="article:published_time")
if pub and pub.get("content"):
    meta["published"] = pub["content"].strip()
# ↑ Webflow 站没这个 meta → meta["published"] 永远是 ""

但 Step 1 的 listing.json 里有日期（May 5, 2026，从列表页抓的）。两端数据没贯穿，agent 必须自己想到去回填，不然博文 date 就被错记成运行当天。

③ Step 7 mark-seen 过度标记

# 用户只挑了 1, 3, 5 三篇渲染。如果直接复用 Step 1 命令做 mark-seen：
$ python3 fetch_listing.py --limit 3 --mark-seen
# 9 个 URL 全被 seen → 用户没挑的 6 篇下次再也不会出现

正确做法是窄化到本次实际渲染的 sources + limit 范围。

修订：4 处 SKILL.md 增量

不动任何脚本，只改文档。SKILL.md 153 行 → 202 行。

改动 1：Step 1 加 --sources 子集示例 + 强制保留 listing.json

- python3 fetch_listing.py --limit 3
+ python3 fetch_listing.py --limit 3
+ # 只抓单一来源
+ python3 fetch_listing.py --sources claude --limit 1
+ python3 fetch_listing.py --sources google,openai --limit 5
+
+ - **重要**：把这一步返回的 JSON **完整保留下来**（例如写到
+   `/tmp/listing.json`），后续 Step 4 需要回填日期，不要丢。

改动 2：Step 2 加豁免出口条款

- ### Step 2：摘要确认（**禁止跳过**）
+ ### Step 2：摘要确认（默认必做，可被显式豁免）
+
+ **例外（豁免条款）**：如果用户在当前请求中已经显式表达了豁免意图——
+ 例如"不用问我"、"全部保留"、"直接整理"、"不要打断"、"自动选最新 N 篇"——
+ 则**跳过本步的反向提问**，直接按用户意图进入 Step 3。但仍需在最终汇报里
+ 明确列出"本次选中的文章"清单（标题 + URL）。

改动 3：Step 4 新增 4.1 元信息回填子节

+ #### 4.1 元信息回填（**关键步骤，容易漏**）
+
+ `fetch_article.py` 输出的 `published` 字段经常为空（典型如 Claude/Anthropic
+ Blog 的 Webflow 站点不带 article:published_time meta）。直接喂给 render_post，
+ to_hugo_date("") 会回退到 datetime.now(UTC)，**博文 date 会被错记成"运行那天"**。
+
+ ```python
+ import json
+ art = json.load(open('/tmp/<slug>.json'))
+ listing = json.load(open('/tmp/listing.json'))
+ match = next((e for s in listing['result'].values()
+               for e in s if e['url'] == art['url']), None)
+ if match and not art.get('published'):
+     art['published'] = match['date']      # 例如 "May 5, 2026"
+ json.dump(art, open('/tmp/<slug>.meta.json', 'w'),
+           ensure_ascii=False, indent=2)
+ ```
+
+ 然后 Step 5 用 `--meta /tmp/<slug>.meta.json`（不是原始 .json）。

改动 4：Step 7 改写为窄化 mark-seen 实例

- 构建无误后再回头执行：
- python3 fetch_listing.py --limit 3 --mark-seen
+ 构建无误后再回头执行 mark-seen，但**不能直接复用 Step 1 的命令**——
+ --mark-seen 会把当前 listing 返回的所有 URL 都标记成 seen，包括用户在
+ Step 2 没圈选的那些；那些没渲染的下次就再也出不来了，等于丢文章。
+
+ # 例 1：用户挑了 google 第 1 篇 + claude 第 2 篇
+ python3 fetch_listing.py --sources google --limit 1 --mark-seen
+ python3 fetch_listing.py --sources claude --limit 2 --mark-seen
+
+ # 例 2：单源单篇（最常见）
+ python3 fetch_listing.py --sources claude --limit 1 --mark-seen
+
+ # 例 3：用户回复"全部"，三家全保留
+ python3 fetch_listing.py --limit 3 --mark-seen

升级断言：从 8 条到 12 条

新增 4 条：

NEW_ASSERTIONS = [
    # ⑨ agent 行为类
    {"id": "exemption-honored",
     "must_not_contain": ["请回复要保留的编号", "请确认编号"],
     "must_contain_any": ["选中", "挑选", "已选", "本次选中"]},

    # ⑩ 产物特征类
    {"id": "date-not-fallback",
     "check": "post.md 的 date 不等于运行当天日期"},

    # ⑪ 状态机类
    {"id": "seen-narrow",
     "check": "len(seen.json) == 用户实际圈选的文章数"},

    # ⑫ 元断言（直接测文档自身）
    {"id": "skill-doc-has-fixes",
     "must_contain": ["豁免", "回填", "窄化"]},
]

第 ⑫ 条是个元断言——直接 grep SKILL.md 文档自身。这条听起来像作弊，但它是唯一不依赖 agent 临场行为的客观度量：能直接告诉你"约束有没有从 agent 大脑迁移到文档里"。

iter-1 vs iter-2 对照

# 断言 iter-1 iter-2
①~⑧ 硬底线 PASS×8 PASS×8
⑨ 豁免行为正确 PASS（agent 临场判断） PASS（按文档照做）
⑩ date 不回退 PASS（agent 手工回填） PASS（按文档照做）
⑪ seen.json 窄化 PASS（agent 自己想到） PASS（按文档照做）
⑫ SKILL.md 含 P0 三条指引 FAIL PASS
总计 11/12 12/12

12 条里 11 条都不变，唯一变化是元断言。这恰好是 Skill 工程化的本质进步：⑨⑩⑪ 在 iter-1 全 PASS 是因为高水平 agent 临场把 SKILL.md 没写明的事做对了，不是 Skill 本身保证的。把约束写进文档之后，⑨⑩⑪ 不再依赖 agent 智力——任何 agent 照做都能 PASS。

五、5 条工程经验

1. 先写 SKILL.md，再写脚本

反例：本能地"上来就 curl + bs4"，写到一半发现没想清楚 Step 2 的用户参与点应该出现在哪里。

修复：先用 SKILL.md 把 7 步骨架画完——把每一步的"输入/输出/触发条件/中断点"全部确定下来——再去写每一步对应的脚本。SKILL.md 是流水线的契约，脚本只是契约的实现。

2. 脚本动作要"零判断空间"

反例：让 fetch_article.py 自己猜 slug、自己决定要不要带来源前缀、自己挑 cover_image。

修复：把所有判断踢给上层（agent 或用户）。fetch_article.py 只接受外部传入的 slug，不做猜测。这条原则的好处是脚本测试简单——给定输入永远输出同样结果，没有隐藏状态。

3. agent 决断只放在两个地方

  • 模型最擅长的地方：自然语言生成（中文复述）
  • 用户必须参与的地方：选择题（摘要确认）

中间环节（图片下载、文件命名、front matter 拼装）全部脚本化。让 agent 干这些事，效率低且不稳定。

4. 评测客观断言只能测硬底线

8/8 vs 8/8 的"打平"假象就是这个原因。客观断言能验证产物形式合规，但抓不到产物质量差异（清洗干不干净、slug 撞不撞、状态机有没有）。

修复：机检 + transcript 对照并行。客观打分用断言；定性差异看两份 transcript 逐段 diff。两个一起看才完整。

5. 元断言比看起来有用

iteration-2 的关键不是修代码，是把 3 条隐性约束写进 SKILL.md。这种改动用 ①~⑪ 这种"测产物"的断言抓不到——产物在 iter-1 已经做对了。元断言（直接 grep 文档自身）是唯一能客观度量"约束是否进入了文档"的方式。

进阶：未来还可以加"transcript 行为类"元断言——比如"agent 是否在最终回报里引用了 SKILL.md 第 X 节作为决策依据"。这一类断言能反映 agent 是不是真的"按文档做"，而不是"碰巧产物对了"。

六、覆盖范围与未做的事

本轮评测覆盖：

  • 1 个 prompt × 2 个 agent variant（with/without skill）× 2 次 iteration
  • 手工亲跑（subagent 工具集只读，无法跑端到端）
  • 真实联网（双跑用的 Claude Blog 文章是 live 抓的，两次抓到的还不是同一篇）
  • 隔离仓库副本，未污染主仓库

没覆盖的部分：

  • 没跑 Google Blog / OpenAI Blog——fetch_listing.py 的 DATE_RE_LONG 没加 re.IGNORECASE，遇到全大写月份（Google 风格的 MAY 4, 2026）会解析失败、回退到 UTC now。这是 P1 已知问题，下一轮再修。
  • 没跑触发评测（run_loop.py 依赖 claude CLI 和 anthropic SDK，本机没装）。
  • baseline 是熟练 agent 跑的，没换"陌生 agent"重跑——理论上陌生 agent 在 baseline 下产物质量会更差，但 with-skill 应该仍然稳定。这是下一个值得做的对照实验。

完整评测产物（断言定义、grading.json、transcript.md、benchmark.json、SKILL.md diff）都落在 .codebuddy/skills/tech-blog-digest-workspace/iteration-{1,2}/，可直接对照本文复现。

题外话：写本文时为了拿一篇 Claude Blog 文章作为评测的"真实数据"，让 Skill 自己抓了一篇 《Deploying Claude across financial services》 — 这篇博客今天也在仓库里。某种意义上，这个 Skill 已经替它自己证明过价值了。
